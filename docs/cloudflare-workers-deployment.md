# Cloudflare Workers Deployment

## Decision

Production runs as a Cloudflare Worker with static assets, an API Worker, Cloudflare D1 storage, and Workers AI embeddings.

The local development server remains unchanged:

- Vite serves the React client.
- Express serves `/api/*`.
- local Transformers.js and `.data/vector-store.json` are used for local semantic search.

The production Worker path exists separately in `worker/src/index.ts` because the existing Express server depends on Node server APIs and filesystem persistence. Keeping a Worker-native entrypoint avoids edge-runtime compatibility risk while preserving the shared API contract used by the React client.

## Cloudflare Bindings

`wrangler.toml` defines these bindings:

- `ASSETS`: serves `dist/client` after `npm run build:client`.
- `DB`: D1 database for search jobs and cached vectors.
- `AI`: Workers AI binding used to generate semantic embeddings.

The Worker uses `@cf/baai/bge-base-en-v1.5` by default, configured with `WORKERS_AI_EMBEDDING_MODEL`. The model returns 768-dimensional embeddings, so D1 stores vectors as JSON arrays with the model name and text hash. If the model changes, cache keys include the new model name and embeddings are regenerated.

## D1 Schema

The first migration is `migrations/0001_cloudflare_worker_storage.sql`.

It creates:

- `vector_records`: cached embeddings for business profiles and tenders.
- `search_jobs`: durable progressive search job state returned by `GET /api/search/:searchId`.
- `browser_sessions`: anonymous browser sessions generated by the frontend and stored in browser `localStorage`.
- `onboarding_profiles`: website and LinkedIn URLs collected during onboarding, keyed by browser session.

D1 replaces two local-only storage mechanisms:

- process memory search jobs in `server/src/services/searchJobManager.ts`
- filesystem vector cache in `.data/vector-store.json`

This makes polling work across Worker isolates and deployments.

There is no user authentication yet. The production app distinguishes users with a stable browser session id generated in the browser with `crypto.randomUUID()` and stored under `tenderDiscoveryBrowserSessionId` in `localStorage`. This is not a security boundary; it is only a lightweight way to group onboarding data and searches from the same browser.

## Runtime Flow

1. `POST /api/search` validates `businessSpecification`.
2. `POST /api/profile` upserts the browser session, company website, and LinkedIn URL when onboarding completes.
3. The Worker fetches current tenders through the shared `FindTenderClient`.
4. Active tenders are written to a D1-backed search job with `embeddingStatus: "pending"` and the browser session id.
5. The response is returned immediately.
6. `ctx.waitUntil()` continues embedding the business profile and tenders in batches.
7. Each batch updates the D1 `search_jobs` row.
8. The client polls `GET /api/search/:searchId` until the job is complete.

This preserves the current progressive UI behavior while avoiding long blocking HTTP requests.

## CI/CD

`.github/workflows/deploy-cloudflare.yml` runs on pushes to `main` and manually through `workflow_dispatch`.

The workflow:

1. installs dependencies with `npm ci`
2. runs `npm run typecheck`
3. builds the client assets
4. applies D1 migrations with `npx wrangler d1 migrations apply DB --remote`
5. deploys with `npx wrangler deploy`

The GitHub repository must define these secrets:

- `CLOUDFLARE_ACCOUNT_ID`
- `CLOUDFLARE_API_TOKEN`

The token needs enough Cloudflare permissions to deploy Workers, apply D1 migrations, use Workers AI, and manage the configured custom domain route.

## Domain Setup

The domain must be an active zone in the same Cloudflare account.

After the exact hostname is known, uncomment the `[[routes]]` block in `wrangler.toml` and replace the placeholder with the Cloudflare-registered hostname:

```toml
[[routes]]
pattern = "app.example.com"
custom_domain = true
```

Use a subdomain such as `app.example.com` unless the root domain should be dedicated entirely to this app.

## First Deploy Checklist

1. Authenticate locally with `npx wrangler login`, or configure CI secrets.
2. Create the D1 database:

```bash
npx wrangler d1 create gtm-hack-tender-discovery
```

3. Replace `REPLACE_WITH_D1_DATABASE_ID` in `wrangler.toml` with the returned database ID.
4. Configure the custom domain route in `wrangler.toml`.
5. Apply migrations:

```bash
npx wrangler d1 migrations apply DB --remote
```

6. Deploy:

```bash
npm run deploy
```

## Operational Notes

Workers AI replaces local Hugging Face Transformers only in production. This avoids bundling model files into the Worker and avoids depending on Worker filesystem APIs.

D1 stores embeddings as JSON because the app computes cosine similarity in application code today. If the tender volume grows, the next storage upgrade should be Cloudflare Vectorize or another vector index, with D1 kept for job metadata and tender snapshots.
